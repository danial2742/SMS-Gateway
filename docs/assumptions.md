# Assumptions

Every assumption made while turning an intentionally open problem statement into a concrete design. Stated explicitly because an undocumented assumption is a landmine for whoever picks this system up next — if any of these turn out to be wrong, the entry below says which part of the design would need to change, not just that it would.

## Scope assumptions

1. **Authentication and tenant management are out of scope.** Every service trusts an already-resolved `tenant_id` in request context; no login, token issuance, or identity provider integration exists in this design. **Impact if wrong:** an auth layer sits entirely upstream of the API Service and resolves to the same `tenant_id` contract — no change to any service downstream of request ingress.
2. **The Operator API is a black-box HTTP endpoint**, abstracted behind an `OperatorClient` interface. Its own rate limits, SLA, and idempotency support are unknown and are not modeled beyond "it can time out or return 5xx/4xx." **Impact if wrong:** if the operator supports a native dedupe key, the at-least-once gap in [queue.md](queue.md) (worker crash between operator call and status commit) can be closed rather than merely mitigated — noted as a deferred improvement below.
3. **Single region for v1.** Consistent with no multi-region requirement in the prompt. **Impact if wrong:** multi-region requires revisiting the single-Postgres-primary-per-shard model into a per-region-primary model with an explicit tenant-to-region assignment story — see [scalability.md](scalability.md) Future scaling options.

## Business-rule assumptions

4. **Every SMS costs a fixed number of credits per message, set by pricing configuration rather than hardcoded.** The schema (`sms.cost`, `batches.unit_cost`) supports per-tenant or per-tier pricing, but pricing logic itself (rate cards, volume discounts) is out of scope — the design assumes cost is resolved to an integer credit amount *before* the atomic deduction, not computed inside it. **Impact if wrong:** none on the atomicity mechanism ([decisions.md](decisions.md) ADR-008), which only requires a resolved integer cost, not a specific pricing formula.
5. **All SMS are single-page.** The design assumes content length validation at intake rejects over-length messages rather than silently segmenting them into multiple pages with per-segment billing. Persian/English cost parity is treated as a *pricing* rule, not an encoding claim — GSM-7 vs. UCS-2 encoding still affects the technical single-page character limit at the operator protocol level, but that limit is assumed enforced at intake, not treated as a billing concern. **Impact if wrong:** multi-page/concatenated SMS would need a segment count factored into `cost` calculation and a `segment_count` column — a schema and pricing change, not an architecture change.
6. **Delivery confirmation (DLR) is out of scope.** `sms.status = SENT_TO_OPERATOR` is treated as the terminal success proxy; there is no ingestion of actual carrier delivery receipts. **Impact if wrong:** this is the single largest deferred feature — see Deferred Features below.
7. **The Operator API is treated as unreliable, not reliable** — the design explicitly assumes it can time out, return 5xx, or return 4xx for invalid destinations, and builds retry/backoff/DLQ/refund around that assumption ([queue.md](queue.md) Retry Queue, [decisions.md](decisions.md) ADR-012). We do *not* assume operator-side idempotency on retried calls, which is the source of the accepted at-least-once delivery gap (see assumption 2).
8. **A permanently failed, paid-for message is refunded automatically.** Not stated in the prompt; assumed as the only defensible behavior for a paid product — silently keeping payment for undelivered service has no credible justification. **Impact if wrong:** if refund should instead require manual review/approval, the DLQ processor's refund trigger becomes a refund *request* awaiting operator action instead of an automatic credit — a workflow change, not a data-model change.
9. **Customers (tenants) are uniquely identified** by a stable `tenant_id`, with no sub-account or multi-user-per-tenant model. **Impact if wrong:** a sub-account model would need its own identity layer above `tenant_id`, with wallet/rate-limit scoping decisions (per-account vs. per-tenant-aggregate) made explicitly — currently unaddressed because unscoped.
10. **Batch recipient lists are bounded** by a configured maximum (e.g. low tens of thousands) per request, to keep transaction and payload size sane. Larger sends are the client's responsibility to split into multiple batch calls, each independently atomic. **Impact if wrong:** raising the bound is a config change up to the point where a single transaction's WAL write size becomes a concern; beyond that it's a batch-of-batches client pattern, not a server-side redesign.
11. **Express SMS volume is a minority of total traffic** (assumed under 5-10% of the 100M/day target) — this justifies sizing Express as a smaller, dedicated pool rather than the default path (see [scalability.md](scalability.md) Capacity planning). **Impact if wrong:** if Express is the majority, capacity-planning numbers (partition counts, worker pool sizing) change, but the architecture — full physical isolation ([decisions.md](decisions.md) ADR-005) — does not; isolation was never premised on Express being small, only capacity sizing was.
12. **"Report" reads are eventually consistent.** Reports are served from a read replica or async-updated aggregate; a report generated milliseconds after submission may not yet reflect the very latest in-flight state. Assumed acceptable since no strong-consistency requirement was stated for read paths. **Impact if wrong:** a strong-consistency reporting requirement would force report reads onto the primary for at least the affected query shape, reintroducing exactly the OLTP/reporting contention [scalability.md](scalability.md) Read replicas is designed to avoid — a real trade-off, not a free change.
13. **Recipient numbers are valid E.164 at intake**, with format validation as the only correctness check performed by this system — no carrier lookup, number-portability check, or live reachability probe before acceptance. **Impact if wrong:** deeper validation would be a pre-acceptance step (its own latency/cost trade-off) or left as-is with invalid numbers surfacing as non-retryable operator 4xx failures (current behavior).

## Deferred features (explicitly out of scope for v1)

These are known-valuable, deliberately not built — listed here so "not present" reads as a decision, not an oversight.

| Feature | Why deferred | Sketch, if built |
|---|---|---|
| **Delivery Receipts (DLR)** | Requires operator webhook ingestion and a correlation mechanism (`operator_message_id`) not exercised by v1's synchronous accept-and-forget model | Ingest operator DLR webhooks, correlate via `operator_message_id`, transition `sms.status` to a true `DELIVERED`/`UNDELIVERED` rather than today's `SENT_TO_OPERATOR`-as-proxy |
| **Scheduled SMS** | Adds a distinct "charge at schedule time vs. charge at trigger time" design decision with its own trade-offs, not resolved by anything in v1 | `scheduled_for` column on `sms`/`batches`; a scheduler polls due rows and performs the same atomic-deduct-and-enqueue path at trigger time |
| **Multi-Operator Routing** | No stated requirement for operator failover or cost-based routing | Abstract `OperatorClient` behind a router selecting by cost/latency/health per destination |
| **Smart Retry (circuit breaker per route)** | Fixed retry budgets (§ [queue.md](queue.md)) are sufficient at assumed single-operator scale | Circuit-breaker per operator route — stop retrying against a route with elevated recent failure rate |
| **Dynamic/tiered priority beyond Express-Normal** | No stated requirement for more than two SLA tiers | Extend DRR into weighted quanta per contracted SLA tier (gold/silver/bronze) |
| **DLQ replay tooling** | Operational tooling, not core delivery logic | Operator-facing UI/CLI to inspect and manually replay DLQ messages after root-causing an outage |
| **Operator-side idempotency key** | Contingent on the real operator API's capabilities, which are unspecified per assumption 2 | Would close the at-least-once delivery gap in [queue.md](queue.md) if the operator supports a dedupe key on ingestion |

See [decisions.md](decisions.md) for the ADRs these assumptions feed into, and [scalability.md](scalability.md) Future scaling options for the infrastructure-scaling (as opposed to feature) roadmap.
